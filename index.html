js中如何实现私有成员

真正的私有化

对于js而言，要实现货真价实的私有变量（外部完全无法访问），只能使用闭包。比如:

闭包大法虽好，但是这样写有一个缺点——每次创建一个实例的时，都要新建相关的method。对于需要大量创建的对象来说，其开销就无法令人满意了（一般情况下，method都是放在prototype上以便所有实例共享使用）。

折中方案

使用prototype的缺点则是将所有成员都暴露了出来。面对这种两难的局面，一般有以下几种折中处理方案。
第一种，也是最简单直接的，约定所有以’_’开头的即为私有属性，客户程序不应该调用它。

当然，如果遇到不遵守约定的人，私有性则会被破坏。

第二种，利用Symbol隐藏私有属性名称，使外界无法直接引用。

Symbol方案已经比较接近完美了，美中不足的是，外界仍然可以通过Object.getOwnPropertySymbols(foo)获取到Symbol，从而破坏私有性。另外必须使用this[symbolName]比不上使用this.name书写顺畅。

第三种，利用WeakMap隐藏私有属性。

WeakMap实现的效果跟Symbol类似，而且没有getOwnPropertySymbols泄密的问题，完美。
另外之所以使用WeakMap，是为了置空对象（foo = null）可以被垃圾回收，防止内存泄露。

新特性 Private class fields

WeakMap使用起来还是稍显复杂。官方在es2020终于提出了一个原生解决方案，简单来说就是在class声明中凡是以#开头的，就代表它是私有属性，比如：

这是因为js本质上还是动态语言，在对象上可以随意添加/删除属性。以foo.name为例，js解释器是无法判断name是初始化设置的私有属性还是后来添加的公开属性的，所以必须在名称上加以区别。

VsCode实践总结

前言

虽然VsCode是一款非常强大的编辑器，但是在深度使用过程中也难免会遇到一些体验不好的地方。因此在这里持续做一个列表总结，避免重复踩坑。

插件篇

按需使用问题

截止2020.11月为止，VsCode所支持的插件按需加载方式有：

  1. 全局启用/禁用
  2. Workspace启用/禁用
  3. 插件自定义启用逻辑（比如指定文件类型）

ps: 如果想看当前项目启用了哪些插件，可以使用Show Runing Extensions命令。

目前最大的问题是，Workspace启用/禁用无法共享，这导致我们需要为每一个项目去手动禁用不需要的插件，非常不便。这个问题还未得到官方解决，民间的替代方案可以
参考(无论哪一种都有缺陷)。

chai cheatSheet

谓词

谓词(变量) = 命题。

类型（DataType）判断

名称 别名 说明
a an 支持所有js内置类型判断，以及提供[Symbol.toStringTag]属性的自定义类型判断。
expect('foo').to.be.a('string')
arguments Arguments 是否是arguments对象
expect(arguments).to.be.arguments
instanceof instanceOf 同instanceof运算符
expect(new Date()).be.an.instanceof(Date)
extensible 对象是否是可扩展的
expect({a: 1}).to.be.extensible
sealed 对象是否是封闭的
expect({a: 1}).to.be.sealed
frozen 对象是否是冻结的
expect({a: 1}).to.be.sealed

等式判断

名称 别名 说明
ok 是否truthy value
expect(1).to.be.ok不推荐使用ok，类型自动转换本身就是难以察觉的事情，用显示的等于判断更加严谨。
比如expect(1).to.be.equal(1)
true 是否 === true
expect(true).to.be.true
false 是否 === false
expect(false).to.be.false
null 是否 === null
expect(null).to.be.null
undefined 是否 === undefined
expect(null).to.be.undefined
NaN 是否isNaN
expect('foo').to.not.be.NaN
empty 是否为空
expect([]).to.be.empty
expect('').to.be.empty
expect({}).to.be.empty
空可以简单理解为对象存在但是其length === 0
equal eq ===判断
expect('foo').to.equal('foo')
eql eqls 近似等价于deep.equal
expect({a: 1}).to.eql({a: 1})
区别在于deep.equal会影响chain之后的所有判断

不等式判断

名称 别名 说明
above gt/greaterThan 大于
expect(100).to.be.above(99)
least gte 大于等于
expect(100).to.be.at.least(100)
below lt/lessThan 小于
expect(99).to.be.below(100)
most lte 小于等于
expect(100).to.be.at.most(100)
lengthOf 长度等于
expect([1, 2, 3]).to.have.lengthOf(3)
根据目标对象的length/size属性判断

存在性判断

名称 别名 说明
exist 相当于!== null && !== undefined
expect(1).to.exist
跟ok的问题类似，如果能指明所比较的值，则不用exist
property 属性是否存在/等于某值
expect({a: 1}).to.have.property('a')
expect({a: 1}).to.have.property('a', 1)
1. 可与deep、nested配合支持嵌套结构。
2. property会改变当前目标值，比如expect({a: 1}).to.have.property('a').that.is.equal(1)
ownProperty haveOwnProperty 自身属性是否存在
效果上等价于own.property
ownPropertyDescriptor 属性描述符是否存在/等于某个descriptor
expect({a: 1}).to.have.ownPropertyDescriptor('a')
ownPropertyDescriptor会改变当前目标值
match matches 正则匹配
expect('foobar').to.match(/^foo/)
keys key是否存在
expect({a: 1, b: 2}).to.have.all.keys('a', 'b')
respondTo respondsTo 对象上是否存在method（如果对象是函数，则检测的是其prototype ）
expect(new Date()).to.respondTo('getTime')
oneOf 是否存在于某个指定数组
expect(1).to.be.oneOf([1, 2, 3])

范围/集合判断

名称 别名 说明
include includes/contain/contains 是否包含：
1. include默认使用===比较每一个子元素，嵌套结构判断需要配合deep/nested。
2. include默认会搜索原型链上的属性，配合own可以阻止这一行为。
expect('foobar').to.include('foo')
expect([1, 2, 3]).to.include(2)
expect({a: 1, b: 2, c: 3}).to.include({a: 1, b: 2})
members 子数组判断
expect([1, 2, 3]).to.have.members([2, 1, 3])
within 区间判断（适用于数字/日期）
expect(2).to.be.within(1, 3)
string 子字符串判断
expect('foobar').to.not.have.string('taco')
closeTo approximately 数字是否靠近某个中心值（+/-半径）
expect(1.5).to.be.closeTo(1, 0.5)
finite 是否有限数字，等价于非NaN且非Infinity
expect(1).to.be.finite

函数式判断

名称 别名 说明
throw throws/Throw 异常捕获
1. throw会改变当前目标值
2. 注意wrap，expect(fn).throw()或者expect(() => fn(args)).throw()
3. 只支持捕获Error或者其派生对象
satisfy satisfies 自定义判断，返回truthy值表示真。
expect(1).to.satisfy(function(num) { return num > 0 })
change changes 是否发生了改变。2种应用场景：
1. 判断调用目标函数后，某个指定函数的返回值发生了改变。
2. 判断调用目标函数后，某个对象的属性发生了改变。
expect(add).to.change(getResult)
expect(add).to.change(obj, "total")
change一般很少使用，更加易用方式是直接对运算结果进行判断。
increase 判断调用目标函数后，某个指定函数的返回值增加了
expect(addTwo).to.increase(getVal).by(2)
decrease 判断调用目标函数后，某个指定函数的返回值减少了
expect(subTwo).to.decrease(getVal).by(2)
by 跟在change/increase/decrease后面，判断差值是多少
expect(subTwo).to.decrease(getVal).by(2)

特殊谓词

名称 别名 说明
fail 创建一个错误命题
expect.fail(actual, expected, [message], [operator])
should.fail(actual, expected, [message], [operator])

副词

副词用于修饰谓词（无法单独使用），从而组合出不同的语义。

名称 修饰对象 说明
not all 否定地，得出与原谓词相反的结论
expect({a: 1}).to.not.have.property('b')
deep equal/include/members/keys/property 递归地，常用于object/array数据的对比
expect({a: 1}).to.deep.equal({a: 1})
nested property 嵌套地，其实就是支持object-path /include
expect({a: {b: ['x', 'y']}}).to.nested.include({'a.b[1]': 'y'});
own property/include 自身地，忽略原型链上的属性
expect({toString: () => {}}).to.have.own.property('toString')
ordered members 有序地，要求集合中元素出现的顺序保持一致
expect([1, 2]).to.have.ordered.members([1, 2])
include members 部分地，针对某个子集元素进行断言
expect([1, 2, 3]).to.include.ordered.members([1, 2])
any keys 至少一个
expect({a: 1, b: 2}).to.not.have.any.keys('c', 'd')
all keys 全部满足
expect({a: 1, b: 2}).to.have.all.keys('a', 'b')
lengthOf above/below/least/most/within 长度上
expect([1, 2, 3]).to.have.lengthOf.above(2)
itself respondTo 自身上，当目标是函数时，迫使respondTo从自身开始检测

连词

连词本身不参与逻辑判断，可以添加到任意位置，用来增强代码的可读性。

Web大屏开发思路

等比缩放

一般大屏的特点就是在一屏内展示内容。考虑到投放屏幕的尺寸跟设计稿往往有所差异，因此大屏需要具备一定的自适应缩放能力。
在不考虑跨屏匹配的前期下，大屏的缩放问题跟移动端碎片化屏幕适配问题类似。因此我们可以利用rem来布局。

contain效果

rem布局原理不再赘述，这里讨论一个新的问题。rem虽然能够让元素弹性缩放，但是它是按单边等比缩放的。比如按屏幕宽度，若10rem=clientWidth，那么凡是width=10rem的元素在任意手机上都会占满手机的宽度。但是对于高度height而言就无法统一了。原本刚好铺满一屏的页面，在另一款分辨率不同的手机上，很可能会出现纵向滚动条。对于大部分Web应用而言，这不是问题，但是对于大屏来说就不太友好了。

为了不出现滚动条，且利用rem等比缩放的便利，我们可以模仿背景图片的contain效果——无论图片多大，都会等比缩放至刚好能够在其容器下完整展示的尺寸。以设计稿尺寸为基准，大屏的contain效果如下图所示：

scale缩放

除了rem之外，另外一种实现等比缩放的方法是利用transform: scale(x, y)缩放。算法与rem如出一辙，只不过最终体现为在根容器元素上设置scale比例。
值得一提的是，在利用rem缩放时，可能会遇到：

  1. FontSize受浏览器限制（比如chrome默认最小字体为12px)。
  2. canvas图表不会自动缩放（需要重新渲染）。
    而scale缩放则没有这些问题，能省不少功夫。

在VSCode上构建风格统一的Vue开发环境

说明

工欲善其事，必先利其器。我们选择VSCode这一款强大且对前端开发非常友好的代码编辑器。

VSCode提供了2种配置模式，用户和工作区。用户配置实际上就是全局配置，对所有的文件都有效。而工作区（目录）配置是局部的，只对工作区下的文件有效。
由于不同的项目可能存在不同的风格，因此我们采取更加灵活的工作区配置方式。

代码格式化

以vue开发为目标，我们需要安装vetur、eslint、prettier这三款插件：

  1. vetur是一款集成插件，提供了针对.vue文件的语法高亮、格式化、校验、智能提示等基础功能（依赖eslint、prettier）。
  2. eslint-vscode静态js质量检查（依赖eslint）。
  3. prettier-vscode代码格式化（自带prettier，但是会按照工作区->全局->自身优先级顺序去调用）。

自带的prettier版本落后而且不支持扩展配置，因此最好安装本地依赖。
