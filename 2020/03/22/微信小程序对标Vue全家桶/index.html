微信小程序框架对标Vue全家桶

前言

小程序模仿/借鉴了React、Vue（特别是Vue）的设计理念与开发方式，因此在这里列出一份小程序跟Vue开发的对比表格，方便有Vue开发经验的小伙伴快速理解小程序开发。

Module 模块化开发环境 支持 小程序只支持CommonJS模块，不能引入其它资源。
支持es6 import/export。		 ✔
Webpack.resolve.alias 路径别名 不支持 见app.require，至少可以使用绝对路径 ✔

设置App启动页可以缓解一些问题。		 ✔

vuex 状态管理 扩展支持 见小程序的MobX绑定辅助库 二期
vue-router 路由管理 支持 路由统一在App.json中维护
不过小程序不支持路由守卫，见文章底部坑点描述		 ✔

ivux-data-cache 响应式数据缓存 不支持 缓存可以，响应式不行。 二期
ivux-dict-manager 数据词典 支持 稍加改造即成纯js库 todo
ivux-enum-manager 枚举定义 支持 稍加改造即成纯js库 todo
less-var-to-js-loader Less与js变量互通 不支持 变通方法js中使用CSS变量 todo
vue-flexible-js rem弹性布局 支持 直接使用rpx单位 ✔

vue-echarts 图表组件 支持 需使用echarts-for-weixin ✔
vant 基础组件 部分支持 需使用vant-weapp ✔
vue-quill-editor 富文本编辑组件 扩展支持 需使用editor

JS-SDK 微信接口 支持 小程序能力是强于公众号的，功能只多不少 ✔

实践坑点总结

1. js模块不要与npm模块同名，某些场景下require会失败。

比如在axios.js中import Axios from ‘axios.js’，Axios会undefined。这个跟小程序的寻址逻辑有关系，npm模块查找优先级较低。

2. 小程序不支持Cookie，因此传统基于SessionID的登录会失效。

解决方法是将接口返回的Set-Cookie信息缓存起来，发送时主动添加到header（weapp-cookie就是利用Storage模拟Cookie）。

3. 不支持路由守卫。
解决方案 优点 缺点
onLoad事件中检查 1. 各种场景都能使用。 1. 页面会闪烁。
2. 入侵业务代码，每个页面都要注册onLoad事件（另一种方法是重写Page）。
包装wx导航API，跳转前检查 1. 无入侵。 2. 无页面闪烁。 1. 只支持js导航，对于<navigator>组件导航以及手机前进后退导航无效。
wx.onAppRoute事件中检查 1. 无入侵。 2. 各种场景都能使用。 1. 页面存在闪烁（比onLoad触发时机要晚，如果页面内容非常复杂，可能闪烁会更明显）。
2. 属于隐藏api，后续可能需要替换。
