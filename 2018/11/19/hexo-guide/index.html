安装/运行：

hexo做了哪些事情

  1. 模版解析，按照一定的规则将模版自动解析为html文件；
  2. 文件管理，抽象出了title、category、tag、date等文件描述符，实现文档的分类存储以及分页导航；
  3. 静态资源映射，在模版中通过相对路径来引用静态资源，最终的依赖关系交给hexo处理；
  4. 以上都是generate，另外hexo还提供了deployment功能，一键部署到服务器

hexo的强大之处

  1. 自定义主题，主题实际上包含页面上除了md定义之外的所有部分，意味着你可以任意更改页面展现形式；
  2. 支持多语言，借助hexo插件可以实现主题的国际化，文章也可以分语言存档；
  3. 增强md，通过标签机制可以实现多种引用；
  4. 插件机制，灵活的控制hexo、添加功能、丰富的第三方插件库、自定义插件==

hexo运行原理

模版路径解析

-> scaffold layout，hexo new [layout] 传递的模版，用来设置front-matter（内置页面属性）
-> markdown layout，基于scaffold生成content-html，放在page.content中
-> theme layout，实现特定页面部分（分页、导航），并容纳page.content
-> top layout，顶层模版，实现公共页面部分（header、footer），所有其它主题模版都会插入到它的body位置
-> Final Html，最终得到静态的html文件！

hexo坑点

  1. 缓存清理：hexo对静态资源的管理是有缓存的，如果发现更改不生效，可以运行hexo clean清理db.json以及public文件

  2. 资源映射问题：若使用md的原生语法![_name_](_url)会产生很多相对路径解析不正确的问题（修改root path…）。正确的姿势是，全局图片使用img插件引用；全局js、css在主题template中引用；post资源使用asset_path\asset_img\asset_link引用；

  3. post分类存放：默认new post会在source/_post文件夹下直接新建帖子。如果post数量太多，就不方便管理了。可以在配置中修改new_post_name，利用全局变量生成按时间归集的post。

  4. 内嵌html元素：md中直接内嵌html，会将每个换行符翻译为
    ，影响排版。解决方案为，使用 插件包裹html，阻止hexo解析其中的内容。不过这种方法也有缺陷，如果使用则其内部就不能再使用hexo插件语法了。 5. asset静态文件被渲染为page：打开post_asset_folder后，默认会将asset文件（.html .js .css等）解析为page文档。解决方案为在skip_render中排除这些文件。

    6. todo 引入html文件：hexo本身并不支持，计划编写一个hexo-asset-include插件，以文本方式导入post-asset文件夹下的文件（若以/开头，则相对于source目录）。参考[hexo-include](https://github.com/pirtleshell/hexo-include#readme)以及hexo自带的asset_path插件。 7. todo 持久化链接不能包含中文：[hexo链接持久化终极解决之道](https://blog.csdn.net/yanzi1225627/article/details/77761488) 8. todo asset_img不能调整图片大小：虽说是md格式，但是引用图片是很常见的操作，让md具备一定的图片调整能力还是很方便的。 9. hexo generate遇到报错。原因是md中不能直接使用"{"、"}"，需要使用{%raw%}{%endraw%}来转义。
