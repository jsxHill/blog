图表库选型

目前主流的Web图表库大致有3类：
| 分类 | 代表框架 | 优缺点 |
| – | – | – |
| 按图表类别（柱状图、折线图、饼图…）封装的图表库 | echarts | 高度封装，通过配置参数即可出成果，但是自由度低，框架开发者未考虑到的需求，难以自定义实现。适合可视化专业知识不够，但是需要输出图表的场景。 |

利用yeoman生成项目代码

yeoman是什么

yeoman是一款使用Node开发的项目代码生成工具，支持多种语言（Web, Java, Python, C#, 等等)。

安装&使用generator

自定义generator

yeoman本身不生成任何代码，而是把这项工作交给生成器插件。

创建一个generator

generator本质上是一个node_module，目录结构如下：

generators为生成器目录。app是默认生成器，供命令yo name使用。
router是子生成器，供命令yo name:sub-generator使用。

package.json配置如下：

  1. name必须以generator-开头。
  2. keywords必须包含yeoman-generator。
  3. 发布后如果期望generator能够被yeoman官方收录，description是必填项。
  4. yeoman-generator是生成器的基类，后续都是在其基础上进行扩展开发。
  5. files定义了生成器所在目录（配合yo命令使用）。

task & run loop

generator由一系列task（即generator原型对象上的method）组成，并通过run loop队列执行task。
另外yeoman还定义了task的优先级（通过name匹配），特殊优先级名称及其执行顺序如下：

  1. initializing，初始化方法
  2. prompting，处理与用户交互的部分
  3. configuring，修改以及保存项目配置
  4. default，特殊优先级之外都归入此类
  5. writing，文件写入处理
  6. conflicts，冲突处理
  7. install，依赖安装（npm、bower）
  8. end，结束/清理阶段

可以在prompting中定义交互式命令行菜单（基于Inquirer.js）。

使用问题总结

this.sourceRoot(path)更改模版存放路径后导致模板文件找不到

默认情况下，模板文件目录是generator下的templates目录（跟index.js同级）。比较坑的是，如果使用sourceRoot(path)修改root，其上下文是destination 目录而非generator目录。因此，这个方法应当避免使用（官方已视其为遗留方法）。
获取generator根目录可以使用this.resolved，获取模板文件path可以使用this.templatePath()。

.gitignore无法复制

本地运行好好的generator发布到npm后再使用，结果发现.gitignore文件没有复制，还多了一份.npmignore文件。原来这是npm导致的。npm publish会自动过滤掉.gitignore文件，因此.gitignore根本不会提交到仓库，自然也无法复制。另外如果未提供.npmignore文件，但存在.gitignore文件，npm会copy .gitignore并作为.npmigore上传，因此多了一份.npmignore文件。
解决办法是，将.gitignore重命名为gitignore_copy，然后在generator中处理为.gitignore。

前端单测之mocha + chai

mocha

mocha是一款可以运行于浏览器以及Node环境下的前端单测框架。它能够让你定义case（测试用例）并执行case，最终得到测试报告。
mocha本身不提供断言功能，而是将其视为开放接口。开发者可以自行选择断言库，只要该库支持断言为否时抛出异常对象即可。

编写测试用例

mocha支持bdd|tdd|qunit|exports四种风格。以BDD为例，编写形式为：

it是用例。describe是用例的分类，可嵌套。TDD结构与其类似，只是接口名称不同，describe() -> suite()，it() -> test()。

TDD与BDD的区别

TDD强调的是对结果进行测试，即传统的功能性测试，将程序视为一个黑盒子，通过对比实际运行结果与预期结果得出结论。而BDD强调的是对逻辑过程进行测试，将需求拆分为故事，然后通过用例来描述故事，相当于是通过技术语言来模拟用户的使用场景，然后通过是否满足验收标准得出结论。BDD听起来很像敏捷开发，事实上也正是如此。之前的TDD只是将测试-开发连接起来，而BDD由于其高可读、直接反映用户场景的描述方式，可以将需求-测试-开发连接起来。

mocha hooks

以BDD为例。mocha允许在describe内定义before(), after(), beforeEach(), afterEach()四类钩子。

它们是按顺序调用的，在所有用例开始之前调用before（只会调用一次），在每个用例开始前调用beforeEach，然后是用例本身it，然后在每个用例结束时调用afterEach，最后在所有用例结束后调用after（只会调用一次）。
需要注意的是，对于嵌套describe场景，内层describe与外层describe的hooks是共享的，每类钩子触发时，会按顺序依次调用已注册的钩子函数（无论来自哪一层）。例如：

chai。

chai是一款支持TDD/BDD两种风格的断言库。

expect/should

chai为BDD提供了expect/should两种方式。
expect用来包装待测试对象，之后就可以进行断言了。

should则是扩展Object.prototype，为每个js对象提供一个should属性作为断言起点。

两者的区别在:

  1. 兼容性不同。expect支持node以及所有现代浏览器，而should支持node以及除IE外的现代浏览器。
  2. 断言起点不同。expect是作为一个包装函数，而should是直接扩展到Object.prototype。
    两者相同的地方在于支持相同的连词（to、be、with…用于增强可读性）与谓词（not、deep、equal…用于逻辑判断）。

chai-as-promise

它会为expect/should/assert添加eventually属性，使其创建基于promise的断言。便于异步测试。

结合Webpack使用

mocha-webpack
mocha-webpack的原理就是以test测试文件为入口，自动调用webpack进行编译打包，然后将输出文件作为mocha的输入，在Node环境下运行测试用例。

其它单测框架

Jest官网

Jest与Mocha相比的话，前者类似一位管家，你只需要发号施令；后者则类似工具箱，要边挑边用。
Mocha的主要优势在于能够和任何js框架/工具无缝衔接，缺点是可能会带来一些初始化成本。
Jest优势在于能够快速搭建可运行、高性能、功能强大的测试环境。缺点在于扩展性相对弱一点，无法支撑某些非React场景（比如Jest并不完全支持vue-loader、不支持webpack的代码分割）。

vue模板继承问题

Vue.extend/Vue.mixin的限制

extend与minxin方法对几乎所有的vue选项都提供了明智合理的合并机制，唯独除了template。template只能选择保留或者覆盖，这对于以继承的方式创建子组件来说很不方便的。保留父组件template意味着无法新增/修改，而覆盖父组件template则会带来严重的代码冗余问题。

通过查阅网上资料Extend Template using Slot、Extend Template using Block、Extend Template，现有的解决方案如下：
| — | — |
| 方法 | 缺陷 |
| — | — |
| 提供多个slot，子组件按条件选用 | 大泥潭，会诞生巨大且难以维护的组件 |
| 继承Parent的同时，引用Parent作为子组件使用 | 代码冗余且容易出bug。因为需要代理Parent的Props、events等等，实例化时还会触发多余的Parent生命周期钩子 |
| 将模版拆分为更小的组件单元，继承时通过component选项替换这些组件单元 | 问题复杂化，无论是编写父组件还是子组件都会变得困难 |
| 像字符串一样拼接template | 明显的代码可读性问题 |
