CSS中颜色的表示方式

CSS颜色

16进制

#2d8cf0{%raw%}

红:绿:蓝 三原色。
三原色添加越多，和成色越亮（淡）。
当三原色数值相等时，始终为灰色，只是明暗程度不等。

RGB/RGBA

rgb(128, 128, 128)
十进制表示形式。

HSL

hsl(60, 100%, 50%)
色调，饱和度，亮度
色调，10进制，从0到360表示色轮上的不同颜色。其中 0 是红色，120 是绿色，240 是蓝色。
饱和度，百分比，0到100。其中 0% 是无色的，100% 是全色的。
亮度，百分比，0到100。其中 0% 是黑色的，100% 是白色的。

weapp核心概念

项目结构

小程序包含一个描述整体程序的app和多个描述各自页面的page。

模块化

小程序采用了CommonJS规范，通过module.exports与require的方式定义/引用模块。

页面生命周期

路由

小程序的路由全部由框架进行管理，即页面栈。对于js逻辑而言，页面栈是只读的。

路由方式 页面栈表现 触发方式
初始化 新页面入栈
打开新页面 新页面入栈 wx.navigateTo
<navigator open-type="navigateTo"/>
页面重定向 当前页面出栈，新页面入栈 wx.redirectTo
<navigator open-type="navigateTo"/>
页面返回 页面不断出栈，直到目标返回页 wx.navigateBack
<navigator open-type="navigateBack"/>
Tab 切换 页面全部出栈，只留下新的 Tab 页面 wx.switchTab
<navigator open-type="switchTab"/>
重启动 页面全部出栈，只留下新的页面 wx.reLaunch
<navigator open-type="reLaunch"/>

WXML

wxml是小程序用来描述页面结构的标记语言，类似HTML，区别在于wxml提供的是语义、功能更加明确的组件标签，并提供模版渲染语法。

ps: <block>仅仅是一个包装元素，类似vue中的<template>，不会被渲染，可接收指令。

Mustache语法

小程序template中使用{{ expression }}来绑定数据，但是expression只支持简单的js表达式:

  1. 绑定变量 {{ variablName }}
  2. 绑定true/false {{ true }}、{{ false }}
  3. 三元运算 {{flag ? true : false}}
  4. 算数运算 {{a + b}}
  5. 逻辑判断 {{a > 1}}
  6. 字符串拼接 {{a + "suffix"}}
  7. 访问对象属性 {{obj.name}}、{{arr[0].checked}}
  8. 创建新数组 {{[name, "john", "smith"]}}，支持展开运算符 {{[...fruits, "apple", "orange"]}}
  9. 创建新对象 {{title: "用户管理", code: "01"}}，支持展开运算符 {{...attrs, title: "系统设置"}}

注意:

  1. 在属性中使用Mustache需要使用””引起来。另外花括号与”之间不要有空格，"{{a}} "会被解析为"{{a + ' '}}"，可能会造成意料之外的副作用。
  2. expression中不支持函数调用，即使是原生方法也不行，如arr.indexOf(“xx”) > 0

page.data用于定义页面数据。修改数据后必须使用this.setData(changedObj, cb)，否则不会更新到视图。

stats.json

stats是webpack生成的模块统计信息。通过它可以分析出模块依赖图，从而优化编译速度。

stats可视化工具

这里试用了几款第三方分析工具（官方本身的太难用，包多一点直接卡死）：

webpack-chart

webpack-chart是一款比较基础的工具，信息检索很不方便，子级模块需要逐层点击展开，操作繁琐且信息展示不全，无法应对module比较多项目。

webpack-visualizer

webpack-visualizer 信息检索能力比webpack-chart要强，每个模块的信息都可以通过鼠标悬浮方式查看，并且还可以分chunk查看。
但是它存在一个bug，就是当chunk中包含非js模块时，其下子模块的actual值跟实际情况不相符。比如：

对于app.xxx.js而言，其内容几乎都来自src下的js模块，因此src的actual实际上是250K左右。
另外，当模块很多时，界面上会出现细小难以操作区域，即使放大浏览器页面也不够。

webpack-bundle-analyzer

webpack-bundle-analyzer是三者之中最好用的统计工具。
它不仅提供了stat-size视图，还有parsed视图（minify之后的大小）、gzip视图（gzip压缩后的大小）。不存在非js模块的错误统计问题。界面支持缩放，任意层级的模块都可以清晰查看。

webpack是如何进行代码拆分的？

chunk拆分

webpack最终会将我们书写代码块合并包到一个bundle里面，作为浏览器的下载单元。因此bundle的大小会影响前端页面的加载性能。
理想的状况是bundle本身足够小(200kb左右)，但是对于现代Web SPA应用而言，巨型bundle是很容易产生的。有效的解决办法就是拆分bundle为多个chunk，再结合前端的动态加载技术实现页面资源的懒加载（按需加载）。

webpack会在3个地方分离代码：

  1. 不同的entry打包为不同的bundle
  2. 通过SplitChunks插件去重和分离chunk
  3. 通过动态导入创建分离的代码

SplitChunksPlugin

SplitChunksPlugin的前任是CommonsChunkPlugin。CommonsChunkPlugin是手动指定需要分离出来的模块，而SplitChunksPlugin的做法更加聪明，它允许开发者制定一组规则，然后在编译过程中将符合规则的模块自动分离出来。

SplitChunksPlugin的默认规则是：

  • 该模块是共享的（被多次引用）或者是来自node_modules文件夹
  • 原始size > 30kb
  • 页面按需加载时，并发请求数 <= 5
  • 页面初始化加载时，并发请求数 <= 3
    若以上规则全部满足，则会发生代码分离。

vue过渡系统

进入离开过渡

设置元素进入/离开时的动效果。
触发点:

  1. 条件渲染 (使用 v-if)
  2. 条件展示 (使用 v-show)
  3. 动态组件 (使用 is)
  4. 组件根节点

transition组件，实现单个元素的过渡

transition组件不会渲染为真实的dom节点。在触发点被激活时，vue会自动嗅探当前元素是否应用了过渡效果，如果是，则会在不同的过渡阶段为触发元素添加不同的过渡class。

过渡周期：

过渡模式mode：

  1. 默认enter与leave同时发生。
  2. in-out：新元素先进行过渡，完成之后当前元素过渡离开。
  3. out-in：当前元素先进行过渡，完成之后新元素过渡进入。

主要场景，多选一过渡
原生标签若名称相同，则需要通过key来区分。多个组件则不用。

transition-group组件，实现多个元素同时过渡

transition-group会被渲染为一个真实dom节点，默认为span。
过渡模式mode不可用，此时不是单个元素的切换。
内部元素总是需要提供唯一key。

transition-group会提供一个新特性move，在元素改变定位时应用，为其添加class v-move。内部实现是利用了FLIP动画队列，让元素平滑的改变位置。

状态过渡

元素本身改变的动效。

  • 数字和运算
  • 颜色的显示
  • SVG 节点的位置
  • 元素的大小和其他的属性
